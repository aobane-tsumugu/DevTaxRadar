# DevTax Radar — 30秒の審査ガイド

## One-line pitch

Claude Code／Codexの定額料金をローカル利用履歴からプロダクト別に配賦し、税務処理候補とその根拠までたどれるようにする、収益化前の個人開発者向けローカルWebアプリです。

## Problem

定額AIサブスクでは、同じ月額の中に新規ソフトウェア開発、既存製品の機能追加、保守、一般学習、私用が混在します。請求書だけでは「どのプロダクトへ、いくら、なぜ配賦したか」を説明できず、通常経費、取得価額、資本的支出等を検討する基礎資料が作りにくいことがペインです。

## Solution

1. Claude Code／Codexのローカル履歴をread-onlyで走査
2. 本文を保存せず、Provider、月、モデル、トークン内訳、ハッシュ化識別子を正規化
3. 作業フォルダをプロダクトと作業目的へ割り当て
4. Providerごとの月額を、Provider内の加重利用量で配賦
5. 配賦結果を3グループと詳細候補で表示
6. 適用ルール、理由、不足事実、匿名化された根拠メタデータまでドリルダウン

## What is novel

- API従量課金ではなく、複数用途が混在する**定額サブスクの配賦**に特化
- ClaudeとCodexのトークン定義を一つの分母へ混ぜない
- 捕捉できないWebチャット・別PC利用をゼロ扱いせず、「未取得利用」として留保
- 私用、未取得利用、丸め調整を含め、配賦合計と月額請求額の一致を保証
- 税務結論を生成するのではなく、候補、根拠、不足情報、確認要否を返す
- 公開デモは合成データ、実利用はローカル完結という二層構成

## Why this should win

- **Impact:** 収益化前でAI支出が先行する個人開発者の、証拠を残せず全額当年費用として扱いがちな実痛に向き合う
- **Originality:** 定額AIの説明可能なプロダクト別配賦から、取得価額・資本的支出等の税務候補までを接続する
- **Technical depth:** 2 ProviderのAdapter、Provider別分母、月額一致の不変条件、SQLite、ローカルAPI、privacy checkを実装
- **Completeness:** 公開デモ、ローカル実履歴走査、複数月、簡略・詳細2画面、根拠ドリルダウン、テストがつながっている
- **Responsible design:** 有利な税務結論を断定せず不足事実を示し、実データをクラウドへ送らず、公開デモは完全な合成データ
- **Future potential:** 共通費、会計連携、複数PC、税理士レビューへ展開できる、AI時代の原価証拠レイヤー

短時間で作ったことを価値の代わりにはしません。画面、履歴取込、計算、不足情報を含む判定、保存、安全な公開、検証を接続した**成果の密度**を評価対象にします。実装根拠は[配賦](../src/core/allocation.ts)、[税務候補](../src/core/taxDecision.ts)、[資産境界](../src/core/assetThresholds.ts)、[Claude Adapter](../src/adapters/claude.ts)、[Codex Adapter](../src/adapters/codex.ts)、[プライバシー検査](../scripts/privacy-check.ts)から確認できます。

## Tax model

画面は判断負荷を下げる3グループ、内部は詳細候補という二段構成です。

| 簡略表示 | 詳細候補の例 |
|---|---|
| 今年の必要経費 | 通常経費 |
| 翌年以後へ残る原価 | 新規ソフトウェア取得価額、資本的支出、制作原価、前払費用 |
| 対象外・要確認 | 私用、未分類、事実不足 |

### 10万円境界をどう扱うか

「AIサブスクの一行が10万円未満だから即時費用」とは判定しません。配賦した直接費等を**資産単位で集計した取得価額**について、供用状況とともに次の候補を案内します。

- 10万円未満: 供用した年の全額必要経費候補
- 10万円以上20万円未満: 通常償却または3年間の一括償却候補
- 20万円以上: 通常の減価償却候補

100,000円ちょうどと200,000円ちょうどを「未満」に含めません。また、この20万円境界と修繕費・資本的支出に関する形式基準を混同しません。青色申告者向け特例は、申告区分、所得区分、取得・製作日、事業者要件、年間上限等を満たす場合だけ候補へ追加します。

## Safety and explainability

- 履歴走査、分類、保存は端末内で完結
- サーバーは`127.0.0.1`だけで待受
- telemetry、クラウド同期、外部LLM送信なし
- プロンプト、応答、ソースコード、元の絶対パスを保存しない
- 識別子とパスは端末内saltでハッシュ化
- 公開物はprivacy checkで実ログ、DB、ホームパス、UUID等を検査
- 公開デモは実履歴からのマスキングではなく、最初から完全な合成データ

## 60秒デモ

公開デモ: [https://devtax-radar.pages.dev/](https://devtax-radar.pages.dev/)

1. 複数月のProduct A／B、私用、未取得利用の配賦を見る
2. 「今年どうなる？」で3グループの全体像を確認
3. 「なぜそうなる？」でProvider・月・プロジェクト集計の根拠へ移動
4. 「税務QA」で取得価額、資本的支出、旧版→新版、供用開始を確認
5. Provider別月額と配賦総額が一致することを確認
6. オンボーディングを開き、ローカル版での履歴走査フローを確認

公開デモは合成データだけを表示し、ローカル履歴を走査しません。実データで使う場合はGitHub／Releaseからローカル版を起動します。

## Judge these points

1. **Pain specificity:** 収益化前の個人開発者が抱える、定額AI費用の証拠化という具体的な問題か
2. **Explainability:** 配賦額からProvider、月、プロダクト、利用量、判定理由まで追跡できるか
3. **Accounting integrity:** 未取得利用と端数を隠さず、月額と配賦総額が一致するか
4. **Responsible tax UX:** 有利な結論を断定せず、不足事実と確認要否を明示するか
5. **Privacy by architecture:** 実ログをクラウドへ送らず、公開デモとローカル版を分離しているか
6. **Reproducibility:** GitHubから誰でも起動し、テストとprivacy checkを再実行できるか

## Deliberate non-claims

DevTax Radarは、事業所得／雑所得、税額、申告内容、税務上の資産単位を自動確定しません。資産計上や翌期以後への費用配分が認められることを保証せず、税務相談や税理士業務を代替しません。現在は資本的支出の当年償却額、制作原価、前払費用、旧版残価移転を算定しません。
