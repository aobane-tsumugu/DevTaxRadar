# DevTax Radar 統合プロダクト仕様

Version: 0.2
Date: 2026-07-18
Status: Local-first hackathon MVP / implementation-aligned specification

## 0. 結論

このプロダクトは、次のものとする。

> **Claude Code／Codexの定額料金をローカル利用履歴からプロダクト・開発工程別に配賦し、資産単位の累積原価、供用状況、税務上の金額境界を追跡して、当年の必要経費と翌年以後へ残る原価を説明可能にする、個人開発者向けTax DevOps。**

単なる「仕事60%、私用40%」の家事按分ツールではない。

```text
Claude Code／Codex利用履歴
→ Provider・月ごとの利用量を集計
→ プロダクト・資産・改良計画へ配賦
→ AI以外の直接費を合算
→ 供用開始・販売・契約期間を判定
→ 10万・20万・40万等の境界を適用
→ 当年必要経費・翌年以後残高・対象外を算出
→ 根拠ログと不足情報を提示
→ ユーザーが確認・確定
```

税務処理を自動的に断定したり、税務上有利になるよう開発実態を変えさせたりするものではない。実際の開発行為を、合理的な配賦、継続的な記録、供用開始等の証拠へ変換する。

### 0.1 現在の実装状態

実装済み:

- Claude Code／CodexローカルJSONL Adapter
- ハッシュ化したメタデータのSQLite保存
- Provider・月別の加重トークン配賦と未取得利用
- Provider×月別の請求額、未取得利用率、プロダクト・資産・作業分類の保存
- 年間サマリー、複数月表示、配賦明細、根拠メタデータの2画面UI
- 通常経費、取得価額、資本的支出、制作原価、私用等の決定木
- 10万円・20万円、青色申告者向け少額特例の条件ガイダンス
- loopback限定API、CSRF／Origin検査、privacy check、合成fixtureによるテスト
- Pull Request／`main` pushのGitHub Actions CI
- `v*`タグから自動公開する、`npm install`不要のGitHub Release ZIP

未実装または未接続:

- 税務決定木・資産金額境界の全入力を編集するUIと、ダッシュボードへの完全接続
- CSV出力、候補の確定・修正、直接費、供用開始証跡の永続化
- 資本的支出の当年減価償却額、制作原価、前払費用、旧版残価移転の算定
- Cloudflare公開デモ

Cloudflareは製品利用の前提にしない。GitHubから取得した各利用者が、ソース版をNode.js 24.14以上で`npm ci`、`npm run dev`して使うか、Release ZIPを展開して`npm install`なしで`npm start`できることを配布ゴールとする。

## 1. ターゲット

### 主対象

> 給与所得があり、これから副業プロダクトを収益化したい個人開発者。売上はまだゼロか少額だが、Claude Code、Codexなどへ毎月多額の先行投資をしている。複数プロダクト、保守、調査、趣味利用が混在し、青色申告や本格的な帳簿管理はまだ始めていない。

### 典型的な状況

- Claude CodeとCodexを定額プランで使っている
- 1つのサブスクで複数プロダクトを開発している
- 趣味、一般調査、販促、保守も同じアカウントで行っている
- 請求書には月額しか記載されず、プロダクト別金額が分からない
- まだ収益が少なく、当年費用だけにすると費用の効果を活かしにくい
- 新規ソフトウェア取得価額、資本的支出、修繕費、少額資産の違いが分からない
- 税理士に相談するときの事実・数字・証拠を整理できない

### 対象外

- 法人の高度な原価計算を完全自動化したい企業
- 税額計算や電子申告までを一括で完了したい利用者
- ログや証拠なしで任意の経費割合を作りたい利用者

## 2. 提供価値

### 中心メッセージ

> **AIサブスクの「何に使ったか」を、税務で説明できる数字に。**

### サブメッセージ

> Claude CodeとCodexの利用履歴から、定額AIサブスクをプロダクトと開発工程ごとに配賦。通常経費、新規ソフトウェア取得価額、資本的支出、制作原価、私用を整理し、未来の売上に対応するAI開発原価を残す。

### このプロダクトが解く問題

API従量課金はリクエストごとの利用額を取得しやすい。一方、定額サブスクは支払額が固定され、同じ契約内で複数プロダクト、保守、調査、趣味を行うため、請求書だけでは配賦できない。

DevTax Radarは、ローカル履歴に含まれる次の情報を使って、月額料金の配賦根拠を作る。

- セッション日時
- 作業フォルダ
- セッションID
- モデル
- 入力、出力、キャッシュ、推論トークン
- ユーザーが登録したプロダクト・資産・開発段階

## 3. 設計原則

### 3.1 ローカルファースト

- 履歴解析と分類はローカルPCで行う
- プロンプト本文、応答本文、ソースコード本文は初期設定で収集しない
- MVPで利用するのは日時、パス、Git情報、モデル、トークン等のメタデータ
- 外部送信やクラウド同期はMVP対象外

### 3.2 説明可能性

すべての金額は次の情報までドリルダウンできる。

```text
月額請求
→ Provider月次分母
→ プロダクト配賦率
→ セッション
→ 作業フォルダ・ブランチ
→ 税務候補
→ 適用ルール
→ 不足情報
→ ユーザー確定
```

### 3.3 事実優先

- 金額境界を超えるための追加開発を推奨しない
- 同一機能を形式的に分割する提案をしない
- 供用開始日を税務上都合よく変更させない
- 実態が一体なら同じ資産・改良計画としてまとめる
- 自動判定は「候補」であり、確定にはユーザーの確認を要求する

### 3.4 Providerを混ぜない

Claude CodeとCodexはトークン定義、キャッシュ項目、モデル構成が異なる。生トークンをそのまま合算して共通分母にしない。

1. Claudeの月額をClaude内の利用比率で配賦する
2. Codexの月額をCodex内の利用比率で配賦する
3. 配賦後の円額をプロダクトごとに合算する

技術構成、ローカル保存、配布、匿名化の方針は[TECHNICAL_DESIGN.md](./TECHNICAL_DESIGN.md)を正本とする。

## 4. 利用履歴コレクター

### 製品実行構成

実履歴をブラウザだけで直接読むことはできないため、GitHubから取得した利用者のPCでローカルWebサーバーとして動かす。

```text
127.0.0.1 only
├─ local collector / adapters
│   ├─ ~/.claude/projects（read only）
│   └─ ~/.codex/sessions（read only）
├─ allocation / tax-candidate engine
├─ local SQLite
└─ browser UI
```

- Claude／Codexの元履歴は変更しない
- 解析結果だけをアプリ用ローカルデータへ保存する
- アカウント、クラウドDB、履歴アップロード、telemetryを必須にしない
- Cloudflare等を使う場合は、履歴を含まない紹介画面または完全な合成デモに限定する
- 初期リリースは、このPCで確認済みの履歴スキーマから対応を始め、Adapterとして拡張する

## 4.1 Claude Code

### このPCのMVP取得元

`~/.claude/projects`配下のローカルJSONLを読み取り、次を取得する。

- `timestamp`
- `cwd`
- `sessionId`
- `message.model`
- `message.usage.input_tokens`
- `message.usage.output_tokens`
- `message.usage.cache_read_input_tokens`
- `message.usage.cache_creation_input_tokens`

同じAPIメッセージIDが複数行に現れる場合があるため、`message.id`で重複排除する。

### 将来の安定取得経路

- OpenTelemetryのトークン、推定コスト、アクティブ時間
- Hookの`session_id`、`cwd`、`transcript_path`
- `session_id`でOpenTelemetryとHookを結合する

### 注意

- サブスク利用時の推定ドル額は、実際の定額請求額ではない
- Claudeチャット、Cowork、他PC等の利用はローカルClaude Code履歴だけでは捕捉できない
- 捕捉できない利用は「未取得利用」として分母に残す

## 4.2 Codex

### このPCのMVP取得元

`~/.codex/sessions`配下のローカルJSONLを読み取り、次を取得する。

- `session_meta.payload.session_id`
- `session_meta.payload.timestamp`
- `session_meta.payload.cwd`
- 最終`token_count`の累積利用量
  - `input_tokens`
  - `cached_input_tokens`
  - `cache_write_input_tokens`
  - `output_tokens`
  - `reasoning_output_tokens`
  - `total_tokens`

1セッション内の`token_count.total_token_usage`は累積値であるため、セッションの最終値を利用する。途中値を合算しない。

### 将来の安定取得経路

- CLIラッパーでは`codex exec --json`の`turn.completed.usage`を利用する
- Hookから`session_id`、`cwd`、`model`を取得する

### 注意

Codex Hookの`transcript_path`は取得できるが、公式にtranscript形式は安定インターフェースではないとされている。対話UIのtranscript解析は互換アダプターとして扱い、次を表示する。

```text
取得方法: local transcript compatibility adapter
信頼度: B
注意: Codex更新により再解析が必要になる可能性
```

## 4.3 実請求額

定額料金は利用履歴に含まれないため、Provider・月単位で次のいずれかにより登録する。

- 手入力
- 請求書ファイル取込
- 将来のメール・カード明細連携

MVPは手入力を必須とする。

```text
2026-04 Claude Code  30,000円
2026-04 Codex        30,000円
2026-05 Claude Code  30,000円
...
```

為替換算が必要な場合は、支払時の円換算額をユーザーが入力する。自動為替計算はMVP対象外。

## 5. 複数月配賦エンジン

## 5.1 配賦単位

```text
Provider × 対象月 × プロダクト × 税務単位候補 × 作業工程
```

例:

```text
Claude × 2026-06 × Product A × A-v2 × 新規開発
Codex  × 2026-06 × Product A × 決済機能追加 × 機能向上
Claude × 2026-06 × 私用      × 対象外 × 趣味
```

## 5.2 利用量

MVPではProviderごとに「加重利用量」を計算する。計算式と各内訳を必ず表示し、ブラックボックス化しない。

```text
Provider月次配賦額
= Provider月額
× 対象プロダクトの加重利用量
÷ Provider月内の総加重利用量
```

キャッシュを含むトークン種別は同価値として扱わず、Provider別の重みを持つ。MVPでは次の2方式を切り替え可能にする。

1. トークン内訳比率
2. セッション稼働時間比率

将来は、モデル別API単価を参考にしたコスト相当指数も追加できる。ただし、サブスクの実額ではなく配賦基準であることを表示する。

## 5.3 未取得利用

Claudeチャット、ChatGPT、別PC、履歴削除済みセッション等を捕捉できない可能性がある。次の入力を必須にする。

```text
この月の未取得利用:
○ なしと確認した
○ 約10%
○ 約25%
○ 約50%
○ 不明
```

「不明」の場合は、捕捉したCode利用へ月額の100%を自動配賦しない。未取得利用額を「対象外・要確認」に残す。

## 5.4 分類ルール

初回オンボーディングで、検出された作業フォルダをプロダクトへ登録する。

```yaml
project_root: C:\work\product-a
product: Product A
asset_candidate: Product-A-v2
stage: development
revenue_model: SaaS
```

判定優先順位:

1. ユーザーがセッションへ付けた手動ラベル
2. Gitリポジトリ・作業フォルダの登録ルール
3. ブランチ・Issue・マイルストーンの登録ルール
4. ローカル分類器の推定
5. 未分類

プロンプト本文を使う意味分類はMVPでは行わない。

## 6. 税務分類モデル

## 6.1 内部では軸を分ける

「5つの税務候補」は画面ラベルとして使うが、内部では次を分離する。

### 用途・成果物の軸

1. 通常経費
2. 新規ソフトウェア取得価額
3. 既存ソフトウェアへの資本的支出
4. 制作原価・仕掛品
5. 私用・家事費
6. 未分類・専門家確認

### 支出時期の軸

- 当月分
- 前払費用
- 未払費用

### ライフサイクルの軸

- 調査
- 開発中
- 完成
- 供用中
- 保守
- 機能追加
- 開発中止・仕損じ
- 旧版廃止
- 新版へ残価移転候補

### 条件付き分類

- 開業費・繰延資産候補

前払費用は「何を作ったか」ではなく「いつサービス提供を受けたか」の軸である。年払いAIサブスクはサービス期間へ分けた後、各月の利用をプロダクトへ配賦する。

## 6.2 新規ソフトウェア取得価額候補

次を満たす利用を候補にする。

- 特定のソフトウェア・バージョンへ直接対応する
- 供用開始前の製作に関係する
- 一般学習や趣味ではない
- 他のプロダクトと合理的に区分できる

取得価額はAI配賦額だけで判定しない。

```text
取得価額候補
= AI配賦額
+ 外注費
+ 素材費
+ 直接対応する設備費・経費
+ 供用に直接必要な費用
+ 条件を満たす旧ソフト残価
```

## 6.3 資本的支出・修繕費候補

供用済みソフトウェアへの作業を次のように分ける。

### 修繕費・通常経費候補

- 障害除去
- バグ修正
- 現状の効用維持
- 原状回復

### 資本的支出候補

- 新機能追加
- 機能向上
- 価値増加
- 利用可能期間の延長

資本的支出か修繕費か明らかでない場合の20万円、60万円、前年末取得価額10%等の形式基準は、新規資産の10万円・20万円境界と別ルールとして管理する。

```text
20万円A: 新規減価償却資産の一括償却境界
20万円B: 一つの修理・改良計画に関する修繕費判定
```

同じ数字でも意味が違うため、UI上で根拠ルール名を表示する。

## 6.4 制作原価・仕掛品候補

収益モデルが次の場合に条件表示する。

- アプリ・ソフトの複製販売
- 受託開発
- デジタルコンテンツ販売
- その他、販売目的の制作物

当年に販売された部分と、年末時点で未販売・制作中の部分を分ける。すべてのプロダクトへ常時表示しない。

## 6.5 私用・対象外

- 趣味開発
- 一般的な自己学習
- 私的な調査
- 業務との関係を明確に区分できない部分

AIが自動で「私用」と断定せず、ユーザー確認を要求する。

## 7. 資産単位・供用開始・金額境界

## 7.1 税務上の資産単位

10万円等の判定は、月額請求、AIセッション、リポジトリ単位で機械的に行わない。独立した効用を持つソフトウェア・バージョン・改良計画候補をユーザーと確認する。

例:

```text
Product A
├─ A-v1 新規ソフトウェア
├─ A-v1 決済機能追加計画
└─ A-v2 全面再設計
```

## 7.2 供用開始

少額資産判定や減価償却開始の前提として、実際の供用開始を記録する。

証跡候補:

- 本番デプロイ
- 初回ユーザー利用
- 課金開始
- 正式な業務工程への組込み
- 正式原稿への採用
- Gitタグ
- リリースノート

単なる公開ボタン、テスト出力、形式的な日付入力だけで確定しない。

## 7.3 新規減価償却資産の金額境界

青色申告を前提にしない基本線:

| 取得価額 | 処理候補 |
| ---: | --- |
| 10万円未満 | 供用した年に全額必要経費 |
| 10万円以上20万円未満 | 通常償却、または3年間で各年3分の1 |
| 20万円以上 | 原則、通常の減価償却 |

100,000円ちょうどは「10万円未満」ではない。

### 青色申告者の特例

一定の中小事業者で青色申告をする個人について、取得・製作日、供用日、所得区分等を確認する。**30万円／40万円の新旧基準は取得・製作日で判定し、供用開始日は減価償却等の別条件として扱う。**

- 対象金額の下限: 10万円
- 2026年3月31日以前の取得・製作: 30万円未満
- 2026年4月1日以後の取得・製作: 40万円未満
- 年間取得価額合計上限: 300万円を事業月数で月割り
- 改正後の適用期限: 2029年3月31日
- 対象所得: 不動産所得、事業所得、山林所得
- 雑所得・白色申告へ自動適用しない
- 一括償却資産として選択した資産へ重複適用しない
- 貸付用資産は、主要な事業として行う貸付け等を除いて対象外

ルールは金額だけでなく適用開始日付きでバージョン管理する。

### ソフトウェアの耐用年数候補

- 複写して販売するための原本、または研究開発用: 3年
- その他のソフトウェア: 5年

用途をユーザーと確認し、3年／5年を金額だけで自動確定しない。

### 資本的支出の現在の算定範囲

資本的支出か通常経費かの候補分類は行うが、資本的支出部分の当年減価償却額はMVPでは算定しない。供用日、耐用年数、償却方法等を不足情報として表示する。

## 7.4 旧版から新版への残価移転候補

次をすべて確認する。

```text
□ 大幅な仕様変更による新ソフトウェアか
□ 新版完成後、旧版を利用しないか
□ 旧版と新版の資産境界が説明できるか
□ 旧版残価を別途除却損等として二重処理していないか
□ 移行計画・廃止記録があるか
```

旧版が10万円未満で供用年に全額必要経費となっている場合、残価は0円であり、新版へ移す残価もない。

## 7.5 開発中止・仕損じ

不要となったことが明確な失敗部分を、完成資産の取得価額から除外できる可能性を表示する。ただし自動的に当年費用へ確定せず、次を要求する。

- 中止日
- 中止理由
- 採用しなかった機能
- 今後利用しないこと
- コード、Issue、設計資料等の証拠

## 8. 画面構成

MVPは**2ページ＋オンボーディングモーダル**とする。

## 8.1 オンボーディング

### 税務プロファイル

- 対象年
- 事業所得／雑所得／未判定
- 青色／白色／未申請
- 開業日
- 少額資産特例の利用済額

### Provider

- Claude月額
- Codex月額
- 対象月
- 契約期間
- 未取得利用割合

## 9. 伴走機能

## 9.1 月次伴走

> 7月のCodex利用の8%はプロダクトを特定できません。月次確定前に確認してください。

> Claudeチャット等の未取得利用が「不明」です。捕捉したClaude Code利用へ月額の100%を配賦せず、未取得利用として残しています。

## 9.2 新規資産

> Product Aはまだ正式利用されていません。新規ソフトウェア取得価額候補として、AI費用を開発中原価に集計します。

> Product Aの累積取得価額見込が94,000円になりました。AI以外の直接開発費、資産単位、供用開始予定を確認してください。

## 9.3 既存資産

> Product Bは供用済みです。今回の作業は新機能追加ですか、それとも障害修正ですか？

> 同一の改良計画と考えられるセッションが複数月にまたがっています。1コミットずつ分割せず、同じ計画として確認してください。

## 9.4 供用開始

> 供用開始を登録するには、本番デプロイ、初回利用、正式業務への組込み等の証拠を1件以上選択してください。

> 現在の取得価額見込は94,000円です。10万円未満かどうかはAI費用だけでなく、他の直接費を含む資産全体で判定します。

## 9.5 禁止する案内

- 「10万円を超えるまで開発しましょう」
- 「今年は公開しない方が得です」
- 「同じ機能を別プロダクトとして分けましょう」
- 「99,999円以下に分割しましょう」
- 「開業日や供用開始日を有利な日に変更しましょう」

代わりに、現実のシナリオごとの結果を比較し、事実と証拠の不足を知らせる。

## 12. ハッカソンデモシナリオ

### Step 1: Scan

```text
Claude Code履歴を検出
Codex履歴を検出
4か月分のセッションを読み込み
プロンプト本文は読み込みません
```

### Step 2: Onboarding

- 検出フォルダをProduct A、B、C、私用へ割り当てる
- 各プロダクトの供用状況を選ぶ
- Claude／Codexの月額を4か月分入力する
- 未取得利用割合を入力する

### Step 3: Allocation

大量のセッションが、Provider・月・プロダクト別の円額へ変換される。

```text
2026-06 Claude
Product A 新規開発      71%
Product B 保守          14%
私用                     5%
未取得利用              10%
```

### Step 4: Tax mapping

- Product Aは供用前なので取得価額候補
- Product Bの障害修正は通常経費候補
- Product Bの新機能追加は資本的支出候補
- 私用は対象外

### Step 5: Asset ledger

AI以外の直接費を加え、Product Aの累積取得価額を表示する。

### Step 6: Threshold

10万円・20万円等の境界、供用開始有無、当年費用見込を表示する。

### Step 7: Evidence

配賦額から根拠セッションまでドリルダウンし、説明可能性を示す。

### 最も強い見せ場

> **このPCに残っていた数百セッション・4か月分の履歴が、短時間でプロダクト別AI原価台帳へ変わる。**

## 15. ハッカソン向けピッチ

### 1行

> **Claude CodeとCodexの履歴から、AIサブスクをプロダクト別原価へ。**

### 30秒

> 収益化前の個人開発者にとって、AIサブスクは大きな先行投資です。しかし定額サブスクでは、複数プロダクト、保守、調査、趣味が混在し、請求書だけでは何にいくら使ったか分かりません。
>
> DevTax Radarは、このPCに残るClaude CodeとCodexの利用履歴から、月額料金をプロダクトと開発工程へ配賦します。さらに、新規ソフトウェア取得価額、資本的支出、通常経費、私用の候補を示し、供用開始、累積取得価額、10万円・20万円等の境界まで伴走します。
>
> 数百のAIセッションを、税理士にも説明できるプロダクト別AI原価台帳へ変える、個人開発者向けTax DevOpsです。

### 訴求上の表現

推奨:

- 適切な年度へ原価配分する
- 取得価額・資本的支出の候補を示す
- 合理的で継続的な配賦根拠を作る
- 実態と証拠を整理する
- 青色申告特例ではない一般の減価償却にも対応する

避ける:

- 税金を自動で先送りする
- 必ず節税できる
- こうすれば資産計上できる
- 金額境界を超える／下回るために開発を調整する
- AIが税務判断を確定する
